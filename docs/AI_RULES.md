# AI 编码与审查规则

> **用途：** 编码细节与红绿灯审查的完整规格。工作流入口见 `AGENTS.md`；阶段状态见 `CONTEXT.md`。

## 1. 红绿灯审查系统 (Traffic Light Audit)

> **核心逻辑**：隐性深度思考，显性简洁输出

每次完成代码改动后、运行客观验证前，AI 必须执行红绿灯审查，并按下方格式输出 `🚦 Audit Report`。

### 审查维度（思考层，不逐条输出）

**支柱 I：真实性与上下文**

1. **幻觉检测**：引用的组件、hook、API 方法、文件路径、契约字段是否真实存在？
2. **历史一致性**：是否违背 `AGENTS.md` 迁移边界、mock/real 分层、NAS 桌面空间关系？
3. **用户意图**：是否理解真实需求，而非只完成字面任务？

**支柱 II：健壮性与工艺**

4. **反惰性**：严禁占位实现、无意义的 `TODO` 留空、组件内硬编码假成功
5. **异常路径**：API 失败、空数据、加载中、权限不足、网络超时是否有可读提示？
6. **前端兼容**：目标浏览器与响应式窄屏下是否可用；文本是否在按钮、表格、标题中溢出？

**支柱 III：架构与未来**

7. **熵增控制**：是否重复造轮子？能否复用现有 hook、store、样式分区？
8. **解耦原则**：组件是否只做展示与交互？mock 数据是否集中在 `mockApi/`？组件是否经 `@/api` 访问能力？
9. **可测试性**：关键适配逻辑能否编写单元测试？是否把难以测试的逻辑塞进了 UI？

### 输出格式（严格执行）

```markdown
## 🚦 Audit Report

**总体评价：** [ 🔴 阻断 / 🟡 可通行 / 🟢 完美 ]

**1. 🔴 阻断项（必须修复）:**
* (若无则写：无)

**2. 🟡 优化项（建议暂时忽略）:**
* (若无则写：无)

**3. 👮 指挥官建议:**
* (基于 `CONTEXT.md` 当前阶段给出明确指令)
```

### 分级标准

| 等级 | 含义 | 典型情形 |
| --- | --- | --- |
| 🔴 **红灯** | 必须修复后才能视为完成 | 破坏架构分层；mock 能力写成生产事实；组件绕过 `api.ts` 直连 `mockApi`；安全漏洞；硬编码后端地址或本机路径；回流 Legacy API 耦合或 vendor |
| 🟡 **黄灯** | 可通行，但应记录并在合适阶段处理 | 性能隐患；测试覆盖不足；治理文档未同步；单文件接近或超过规模阈值；主观体验项未验收 |
| 🟢 **绿灯** | 满足需求，无明显风险 | 符合阶段目标，客观验证可继续执行 |

### 与阶段文档的衔接

- Audit Report 中的 🔴 阻断项，若影响阶段推进，应同步写入 `CONTEXT.md` 的「当前阻断项」
- 🟡 优化项中可跨任务跟进的内容，可写入 `CONTEXT.md` 的「剩余黄灯」
- 总体评价为 🔴 时，不得将相关客观项标为完成；须先修复或明确降级方案

### 代码规模触发项

- 单个源码文件超过 **350** 行：红绿灯审查必须说明是否仍保持单一职责
- 超过 **450** 行：继续追加逻辑前必须在审查中评估拆分方案
- 超过 **500** 行：默认视为维护风险；除非是低复杂度映射或静态数据，否则 🔴 或至少 🟡 并给出拆分计划

## 2. 审查与验证顺序

```text
思考（LESSONS.md）→ 编码 → 🚦 Audit Report → npm run verify → 同步 CONTEXT / README / docs
```

- 未跑客观验证，不得把客观项标为完成（见 `LESSONS.md` H-005）
- 主观验收项（NAS 风格、布局密度、交互路径）不由 typecheck/build 代替
